# 多人旅游偏好评分与推荐开发方案

## 目标

构建一个多人旅游偏好匹配工具，用于同学、朋友或家庭成员共同填写旅游兴趣问卷，并基于结构化评分推荐大家都能接受、分歧较小、执行可行的旅游地点和活动项目。

核心原则：

- 不使用简单总分。
- 所有维度统一标准化到 `0-100`。
- 题目数量不同的维度按比例归一化，避免题少的维度天然低分。
- AI 只负责解释、总结和生成方案，确定性算法负责评分和筛选。

## 维度模型

建议将旅游偏好拆成以下维度：

| 维度 | 含义 |
| --- | --- |
| `destination` | 目的地类型偏好 |
| `activity` | 活动体验偏好 |
| `culture` | 人文/研学兴趣 |
| `nature` | 自然风光兴趣 |
| `food` | 美食兴趣 |
| `photo` | 拍照打卡兴趣 |
| `relax` | 休闲放松倾向 |
| `pace` | 行程强度接受度 |
| `physical` | 体力消耗接受度 |
| `budget` | 预算敏感度 |
| `transport` | 交通便利要求 |
| `comfort` | 住宿/舒适度要求 |
| `crowd` | 人流接受度 |
| `freedom` | 自由活动需求 |
| `veto` | 明确排斥项 |

每个问题都应映射到一个或多个维度，而不是只保存答案文本。

## 分值标准化

每个维度独立计算：

```text
维度得分 = 当前维度实际得分 / 当前维度可得最高分 * 100
```

不要使用：

```text
总分 = 所有问题分数相加
```

因为某个维度可能只有 2 个问题，另一个维度有 8 个问题，直接相加会让题目少的维度天然吃亏。

正确做法：

```text
nature_score = nature_actual / nature_max * 100
food_score = food_actual / food_max * 100
culture_score = culture_actual / culture_max * 100
```

最终推荐时再按维度权重合成。

## 题型映射规则

### 李克特五级量表

五级量表直接转分：

| 选项 | 标准分 |
| --- | ---: |
| 1 | 0 |
| 2 | 25 |
| 3 | 50 |
| 4 | 75 |
| 5 | 100 |

例如：

```text
喜欢自然风光：5
nature += 100
```

### 多选题

每个选项映射到一个或多个维度。

例如选择“美食探店”：

```json
{
  "food": 100,
  "relax": 40,
  "destination": 50
}
```

多选题不要简单“选中 +1”，而要按标签权重加分。

### 单选题

单选题也应映射为多维权重。

例如“旅行节奏”：

| 选项 | relax | pace | physical |
| --- | ---: | ---: | ---: |
| 轻松度假 | 100 | 20 | 20 |
| 每天 1-2 个重点 | 70 | 50 | 40 |
| 高密度打卡 | 30 | 90 | 70 |
| 研学式深度体验 | 50 | 70 | 50 |

这样系统知道的不只是“用户选择了轻松度假”，还能得到更可计算的偏好向量。

## 重要性排序

重要性排序建议使用 Borda Count 转成权重。

例如 8 个因素排序：

```text
第 1 名 = 8 分
第 2 名 = 7 分
第 3 名 = 6 分
...
第 8 名 = 1 分
```

再归一化：

```text
weight = item_rank_score / all_rank_score_sum
```

如果小组最看重预算、交通、住宿，推荐算法就应提高这些维度的权重。

## 个人画像

每个人提交后生成个人偏好向量：

```json
{
  "nature": 82,
  "food": 91,
  "culture": 64,
  "photo": 76,
  "relax": 70,
  "pace": 42,
  "budget": 88,
  "comfort": 74,
  "transport": 85
}
```

同时保存排斥项和必须满足项：

```json
{
  "veto": ["高温暴晒", "长时间坐车"],
  "mustHave": ["交通便利", "住宿舒适"]
}
```

## 小组汇总

小组维度不只算平均值，还要计算分歧。

每个维度统计：

- 平均分 `mean`
- 中位数 `median`
- 最低分 `min`
- 最高分 `max`
- 标准差 `std`
- 一致度 `consensus`

一致度可先使用：

```text
consensus = 100 - std
```

推荐时优先选择：

```text
平均分高 + 标准差低
```

即“大家都喜欢，且分歧不大”的地点和项目。

## 人数较少时的置信度

如果只有 2-3 人填写，不应给出过度确定的结论。

可以增加小组置信度：

```text
group_confidence = min(100, 人数 / 6 * 100)
```

示例：

| 填写人数 | 置信度 |
| --- | ---: |
| 2 | 33% |
| 4 | 67% |
| 6+ | 100% |

页面可提示：

```text
当前样本较少，推荐结果仅作初步参考。
```

## 推荐算法

先过滤硬条件：

- 预算不符合：排除或强扣分
- 天数不符合：排除或强扣分
- 有人强烈排斥：强扣分或排除
- 交通方式不现实：扣分

再计算匹配分：

```text
match_score =
  维度相似度 * 0.5
+ 共识度 * 0.2
+ 重要性权重匹配 * 0.2
+ 可执行性 * 0.1
- veto_penalty
```

简单版本：

```text
地点得分 = Σ(小组维度分 * 地点维度分 * 维度权重)
```

输出内容：

- 最推荐地点
- 备选地点
- 不推荐地点
- 推荐活动
- 避雷项目
- 推荐理由

## AI 的位置

AI 不负责原始评分。

算法输出结构化结果：

```json
{
  "topCities": ["成都", "南京", "厦门"],
  "topActivities": ["美食探店", "博物馆展览", "轻松拍照"],
  "avoid": ["高强度徒步", "早起赶路"],
  "reason": []
}
```

AI 负责生成：

- 推荐理由
- 分歧解释
- 三套候选方案
- 行程草案
- 需要进一步确认的问题

这样结果更稳定，也更容易解释。

## 开发阶段

### 第一阶段：多人问卷

- 创建小组
- 生成分享链接
- 每人填写问卷
- 保存 JSON 响应

### 第二阶段：评分模型

- 题目映射维度
- 个人画像
- 小组画像
- 维度标准化

### 第三阶段：推荐系统

- 目的地库
- 匹配算法
- 推荐结果页
- 避雷分析

### 第四阶段：AI 分析

- AI 总结
- 推荐理由
- 行程草案
- 分歧解释

### 第五阶段：管理后台

- 管理题目
- 管理维度权重
- 管理地点库
- 查看历史小组

## 结论

表单必须拆成多个维度，每个维度独立统计，并统一标准化到 `0-100`。不要使用简单总分。这样无论某个维度有 2 个题还是 8 个题，都能公平参与推荐。
